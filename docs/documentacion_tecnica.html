A) Arquitectura del Sistema

PadelPro App sigue una arquitectura monolítica integrada: el backend Spring Boot actúa como servidor web y expone tanto la API REST como los archivos estáticos del frontend a través del mismo puerto (8080). No existe un proceso de build separado para el frontend.

B) Modelo de Datos

La base de datos utiliza H2 en memoria durante el desarrollo, con esquema gestionado automáticamente por Hibernate. El modelo consta de dos entidades principales: jugador y usuario.

Diagrama de Tablas
📋 jugador
PK id_jugador VARCHAR
nombre_completo VARCHAR
nacionalidad VARCHAR(3)
mano_dominante VARCHAR
ranking_fip INT
puntos INT
categoria VARCHAR
posicion_juego VARCHAR
foto_perfil VARCHAR (URL)
fecha_actualizacion_ranking VARCHAR
👤 usuario
PK id_usuario BIGINT AUTO
UQ nombre_usuario VARCHAR(50)
UQ email VARCHAR(100)
password_hash VARCHAR
fecha_registro TIMESTAMP
⚠️ Nota sobre la base de datos
Las tablas no tienen relación directa entre sí ya que la wishlist y el historial de reservas se gestionan en el frontend (localStorage). En una versión de producción se añadirían tablas de relación y se migraría a MySQL/PostgreSQL.
Descripción de Tablas
Tabla Propósito Campos clave Poblado por
jugador Almacena el ranking mundial FIP de jugadores masculinos y femeninos id_jugador (clave compuesta FIP), puntos, ranking_fip, categoria DataInitializer al arrancar + endpoint de sync
usuario Cuentas de usuario registrados en la aplicación nombre_usuario (único), email (único), password_hash Endpoint POST /api/auth/register
Seguridad de contraseñas

Las contraseñas se almacenan con hash SHA-256 implementado en AuthService. Nunca se guarda la contraseña en texto plano. Adicionalmente, las validaciones backend rechazan contraseñas de menos de 6 caracteres y verifican unicidad de nombre de usuario y email.

Datos de sesión (cliente)

La sesión del usuario no utiliza JWT ni cookies de servidor. Tras el login exitoso, el frontend guarda en localStorage el objeto { nombreUsuario, idUsuario } bajo la clave padel_session. Al recargar la página, auth.js lee este valor y restaura la sesión automáticamente.

C) API REST y Endpoints

El backend expone una API REST bajo el prefijo /api. Todos los endpoints devuelven JSON y permiten CORS desde cualquier origen (@CrossOrigin(origins = "*")) para facilitar el desarrollo local.

Autenticación

La aplicación utiliza un sistema de sesión basada en localStorage (sin JWT). El backend no comprueba tokens en las peticiones salvo en el proceso de login/registro. Las secciones protegidas (Wordle, Equipamiento, Wishlist, En Directo, Reservar Pista) son gestionadas por auth.js en el lado del cliente.

Endpoints de Autenticación — /api/auth
Método Ruta Descripción Body Auth
POST /api/auth/register Registrar nuevo usuario. Valida campos, verifica unicidad y hashea la contraseña. { nombreUsuario, email, password } No
POST /api/auth/login Iniciar sesión con usuario o email + contraseña. Devuelve nombre e ID del usuario. { identificador, password } No
POST /api/auth/logout Endpoint de cierre de sesión (la lógica real ocurre en el cliente). — No
Endpoints de Jugadores — /api/jugadores
Método Ruta Descripción Query Params Auth
GET /api/jugadores Lista todos los jugadores ordenados por ranking FIP. Admite filtros opcionales de categoría y nacionalidad. ?categoria=MASCULINO
?nacionalidad=ESP
?limit=100		 No
Endpoints de Administración — /api/admin
Método Ruta Descripción Respuesta Auth
GET /api/admin/sync-ranking-masculino Descarga el PDF oficial del ranking FIP masculino, lo parsea e inserta/actualiza los jugadores en la base de datos. { ok, categoria, actualizados } No *
GET /api/admin/sync-ranking-femenino Igual que el anterior pero para el ranking femenino. { ok, categoria, actualizados } No *
* Endpoints de admin sin protección
En la versión actual los endpoints de administración no requieren autenticación para simplificar el desarrollo. En producción se protegerían con Spring Security y un rol de ADMIN.

Pasos de Instalación en Producción

Paso 1 — Instalar Java 21 en Ubuntu:

sudo apt update
sudo apt install openjdk-21-jdk -y
java -version   # Verificar instalación

Paso 2 — Clonar y compilar:

git clone https://github.com/alejandroferrersfc-cell/padel-App.git
cd padel-App
chmod +x mvnw
./mvnw clean package -DskipTests
# El JAR se genera en: target/padel-backend-0.0.1-SNAPSHOT.jar

Paso 3 — Ejecutar el JAR:

java -jar target/padel-backend-0.0.1-SNAPSHOT.jar
# App disponible en http://<IP-servidor>:8080

Paso 4 (opcional) — Ejecutar como servicio systemd:

# /etc/systemd/system/padelapp.service
[Unit]
Description=PadelPro App
After=network.target

[Service]
User=ubuntu
WorkingDirectory=/home/ubuntu/padel-App
ExecStart=java -jar target/padel-backend-0.0.1-SNAPSHOT.jar
SuccessExitStatus=143
Restart=on-failure

[Install]
WantedBy=multi-user.target

Migración a Base de Datos Persistente (Producción)

Para producción real se recomienda sustituir H2 por PostgreSQL o MySQL. Los pasos son:

  1. Añadir dependencia spring-boot-starter-data-jpa + driver JDBC correspondiente al pom.xml.
  2. Cambiar spring.datasource.url a jdbc:postgresql://localhost:5432/padeldb.
  3. Cambiar spring.jpa.hibernate.ddl-auto de create-drop a update o gestionar migraciones con Flyway.
  4. Configurar credenciales de producción como variables de entorno del servidor.
✅ Ventaja del diseño actual
Al ser un JAR autocontenido (Tomcat + Frontend + API todo en uno), el despliegue es extremadamente sencillo: basta con copiar el JAR al servidor y ejecutarlo. No requiere nginx, node, ni ningún servidor adicional.
